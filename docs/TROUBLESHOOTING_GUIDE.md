# 问题排查指南

## UI/显示问题的诊断技巧

当遇到 UI 或显示相关的问题时，请遵循以下最佳实践：

### 1. 立即提供视觉证据（最重要！）
✅ **第一时间提供截图或录屏**，而不是仅用文字描述
- 截图能消除 90% 的歧义
- 对于 UI 问题，截图是必需的
- 一张截图胜过千言万语

### 2. 使用时间线格式描述问题
按**时间顺序**清晰列出每个步骤：

```
步骤1：[具体操作] → [屏幕显示的内容] → [期望结果] vs [实际结果]
步骤2：[具体操作] → [屏幕显示的内容] → [期望结果] vs [实际结果]
步骤3：...
```

### 3. 区分"代码逻辑"和"视觉效果"
- ❌ 模糊："选择项变了"
- ✅ 具体："屏幕上显示的高亮项从第7个工具变成了第1个工具"

### 4. 提供完整的上下文信息
- 操作前的初始状态
- 详细的操作步骤
- 操作后的实际状态
- 期望的正确状态

### 5. 使用标准的"重现步骤"格式

```markdown
## 问题描述
[一句话概括问题]

## 重现步骤
1. 进入工具选择界面，当前选中工具A（第1个）
2. 按下键选择工具B（第7个）
3. 按 Enter 执行工具
4. 观察倒计时界面

## 期望结果
工具列表中工具B（第7个）保持高亮显示

## 实际结果
工具列表中工具A（第1个）被高亮显示

## 截图
[附上截图]
```

### 6. 明确问题发生的时机
- ❌ 模糊："执行后选择项变了"
- ✅ 具体："按 Enter 后，在倒计时显示期间，屏幕上的工具列表中第7个工具的高亮消失，第1个工具被高亮"

### 7. 先确认最小可重现场景
对于复杂问题，先回答这些问题：
- 第一次执行时是否正常？
- 只执行一次是否有问题？
- 问题是否每次都能重现？
- 问题是否只在特定条件下出现？

### 8. 区分不同类型的"状态"
- **代码状态**：变量的值、对象的属性
- **视觉状态**：屏幕上实际显示的内容
- **用户感知**：用户看到和理解的内容

明确说明问题出在哪个层面。

## 引导性问题模板

当问题描述不清晰时，使用以下问题引导：

1. **"能否提供一张截图，显示问题发生时的完整屏幕内容？"**
2. **"请按时间顺序描述：第1步你做了什么，屏幕显示什么；第2步..."**
3. **"这个'变化'具体是指屏幕上哪个位置的哪个元素发生了什么变化？"**
4. **"问题发生在按下按键之前、按下瞬间、还是之后的某个时刻？"**
5. **"除了这个问题，屏幕上还显示了哪些其他内容？"**
6. **"这个问题每次都会出现吗？还是偶尔出现？"**

## 调试技巧

### 使用调试输出验证假设
在关键位置添加调试输出，确认代码逻辑：

```python
import sys
print(f"[DEBUG] 变量名={变量值}", file=sys.stderr)
```

### 分离关注点
- 先确认代码逻辑是否正确（通过日志）
- 再确认显示逻辑是否正确（通过截图）
- 最后确认两者的同步是否正确

## 案例：工具选择界面问题

### 问题描述（初始，不清晰）
"执行工具后，选择项会自动变为第一个"

### 问题描述（改进后，清晰）
```
重现步骤：
1. 进入工具选择界面，默认选中 [Wsl] Kiro CLI（第1个）
2. 按下键6次，选中 [Win] Kimi CLI（第7个）
3. 按 Enter 执行工具
4. 工具在新窗口启动
5. 观察当前终端的倒计时界面

期望结果：
倒计时界面的工具列表中，[Win] Kimi CLI（第7个）保持高亮

实际结果：
倒计时界面的工具列表中，[Wsl] Kiro CLI（第1个）被高亮

截图：
[附上执行前和倒计时时的两张截图]
```

### 关键发现
提供截图后，立即发现：
- 工具列表仍然显示在屏幕上（不是消失了）
- 高亮位置确实变了（不是索引计算错误）
- 倒计时信息显示在工具列表下方（说明是同一个屏幕）

这些信息帮助快速定位到根本原因：Live 上下文退出时的屏幕状态问题。

## 总结

**最重要的原则：对于 UI 问题，截图是第一优先级！**

如果一开始就提供截图，很多问题可以在几分钟内定位并解决。文字描述再详细，也比不上一张清晰的截图。
